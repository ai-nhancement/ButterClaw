# Plan: First-Run Setup Redesign

**Goal:** Replace the terminal-based setup wizard with a web UI that gets new users to their first conversation in under 60 seconds.

**Problem:** The current `openclaw onboard` wizard asks 15+ questions in a terminal before the user can have a single conversation. Risk acknowledgements, gateway bind modes, Tailscale exposure, workspace paths, skills config, hooks — all before the user knows what the system does. Most new users will give up or misconfigure.

**Principle:** Meet the user where they are. Start simple. Add complexity through the UI as they learn.

---

## The Experience We Want

1. User clones ButterClaw, runs `npm install`, runs `npm run dev`
2. Gateway starts with auto-generated minimal config (if none exists)
3. Browser opens to `http://127.0.0.1:18789/`
4. Instead of the normal chat UI, they see a **single setup page**:
   - Pick a provider (Anthropic / Google / OpenAI / Local)
   - Paste API key
   - Model auto-selects based on provider (sensible default)
   - One "Start" button
5. Config is written. Page redirects to the chat UI. First conversation begins.
6. Everything else (channels, crons, skills, advanced gateway config) is accessible through a Settings page later.

Total time from clone to first conversation: **under 60 seconds.**

## Implementation Plan

### Phase 1: Gateway Starts Without Full Config

**Files to modify:**
- `src/gateway/server.impl.ts` — Allow startup with missing/empty config
- `src/config/config.ts` — Auto-generate minimal gateway config on first run

**What happens:**
- If `~/.openclaw/openclaw.json` does not exist, create it with the bare minimum (port + bind + auto-generated auth token)
- Gateway starts and serves the Control UI
- A flag `needsSetup: true` is added to the bootstrap config when no auth profiles exist

**Key constraint:** The gateway must start and serve HTTP even without a model configured. Right now it may fail or show errors if no provider is set up. We need it to start cleanly and defer model config to the setup page.

### Phase 2: Setup API Endpoint

**New file:** `src/gateway/setup-api.ts`

**Endpoints:**
```
GET  /api/setup/status     → { needsSetup: boolean, hasProvider: boolean }
POST /api/setup/configure  → { provider: string, apiKey: string, model?: string }
```

**POST /api/setup/configure behavior:**
1. Validate the API key format (basic pattern check, not a live call)
2. Write auth profile to config
3. Write model defaults to config
4. Set `needsSetup: false` in runtime state
5. Return `{ ok: true, redirect: "/" }`

**Security:** Only accessible from loopback. Requires the gateway auth token (auto-generated and shown in terminal on first start).

### Phase 4: Bootstrap Flow Wiring

**Files to modify:**
- `src/gateway/control-ui.ts` — Add `needsSetup` to bootstrap config
- `src/gateway/server-http.ts` — Register setup API routes before Control UI catch-all

**Route priority (updated):**
```
1. Health checks
2. Setup API (/api/setup/*)     ← NEW
3. Tinker dashboard (/tinker/)
4. Plugin routes
5. Control UI SPA (/)
```

### Phase 5: Terminal Fallback

**Keep the terminal wizard working** for headless/SSH setups. But change the default behavior:

- `npm run dev` → starts gateway, opens browser to setup page (if first run)
- `openclaw onboard` → still works for terminal-only environments
- `openclaw setup --headless` → minimal terminal prompts (provider + key only)

---

## What We Do NOT Change

- The config file format (`openclaw.json`) stays the same
- The terminal wizard still works for power users
- No existing config is migrated or broken
- Gateway internals are unchanged — we only add the setup API and modify the startup path
- All channel, cron, plugin, and advanced config remains accessible through the existing configure command and future Settings UI
